## Quick orientation

- Current version: **V8.6 — Auto theme follows OS prefers-color-scheme**
- Stack: Next.js 15 App Router · TypeScript · Tailwind v4 · Drizzle + Neon · iron-session · Steam OpenID
- Hard constraint: everything must remain on **free tiers** — no paid APIs, no metered per-item costs
- Transactions are **not available** (neon-http driver) — writes are sequential, partial state is accepted

## Recent changes (V4.6–8.6)

- **V8.6 — Auto theme follows the OS.** New default. Fresh visitors with no theme cookie now get `data-theme="auto"` which resolves to **Default (light)** under `prefers-color-scheme: light` and **V1 (dark)** under `prefers-color-scheme: dark`, matching whatever their browser / OS is set to. Existing users who already picked a theme are unaffected — their cookie keeps winning. Mechanism: `THEMES` gained an `auto` value and `DEFAULT_THEME` flipped to it, so the server renders `<html data-theme="auto">` for cookie-less visitors. A synchronous inline `<script>` at the top of `<head>` reads `matchMedia('(prefers-color-scheme: dark)')` and rewrites the attribute to the concrete value (`v1` or `default`) **before first paint** — so no flash of the wrong palette, and all the existing V5.6 default-theme text-colour remaps kick in correctly (they're scoped to `[data-theme="default"]`, which is what the DOM now carries). The script also subscribes to `matchMedia`'s `change` event, so flipping your OS theme while the tab is open flips the site live without a reload. Belt-and-braces: a CSS `@media (prefers-color-scheme: dark) html[data-theme="auto"] { ... }` block applies the V1 palette even if the script is blocked (strict CSP, noscript) — the remaps won't fire in that edge case, but the base palette will still be dark. `Auto` is also selectable from the theme picker now, so users who opted into `default` or `v1` can move back to OS-driven behaviour.

- **V8.1 follow-up — English-only tag catalogue enforcement.** Same release. Tag *slugs* were already ASCII-only (URL-safety regex in `createTag`/`updateTag`); this release extends the same rule to the human-readable *display name* via `isEnglishTagName` in `lib/i18n/english-tag.ts` (regex `/^[A-Za-z0-9 '\-&()\/,.+]+$/`). Called from both `createTag` (Mod+) and `updateTag` (Creator). Non-Latin scripts, accented Latin (`café`, `Motörhead`), umlauts, and CJK are all rejected server-side with a friendly error. Audited all 50 existing tags before landing — zero retroactive breakage. Locking names to ASCII is the cheapest way to guarantee that. (Slugs are unaffected — they were already constrained.)

- **V7.8 — site-rating sort → net upvote score.** The `site-rating` / `site-least-rating` sort modes used to order by `up / (up+down)` (star ratio) gated behind a min-votes floor, mirroring Steam's own rating semantics.
