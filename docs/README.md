## Quick orientation

- Current version: **V9.8 / BETA 3.0 update — Triage filter remount + deploy guard + Discord-friendly OG card**
- Stack: Next.js 15 App Router · TypeScript · Tailwind v4 · Drizzle + Neon · iron-session · Steam OpenID
- Hard constraint: everything must remain on **free tiers** — no paid APIs, no metered per-item costs
- Transactions are **not available** (neon-http driver) — writes are sequential, partial state is accepted

## Recent changes (V4.6–V9.8)

- **V9.8 — Triage filter actually swaps the stack + local builds can't clobber live deploys + rich OG card for Discord/Twitter.** Three bug-fix-style cleanups on the V9.7 batch. (1) **Triage kind filter was ignored after the first click.** `TriageStack` snapshots the `cards` prop into a `buffer` state on mount and only swaps that buffer in when `index >= buffer.length` (i.e., the mod has swiped through everything). Switching filters via the pills updated the URL and server-re-rendered the page with a new `cards` prop, but since `index` was still well below the old buffer's length the old slice kept showing. Added `key={activeKind ?? "all"}` on `TriageStack` in the page so React unmounts + remounts it with fresh state whenever the filter changes — new cards become the initial buffer instantly, the swipe stack starts from the top of the filtered slice. (2) **`scripts/complete-deploy.ts` now short-circuits outside Vercel.** On a developer's laptop `npm run build` uses the same `DATABASE_URL` as prod (there's only one Neon DB). Every local test build was silently stamping `completed_at` on whichever deploy announcement was pending at that moment, so real visitors would see the countdown briefly tick then snap straight to "New version is live — reloading" because `completed_at` was set before `scheduled_at` had even passed. The script now bails with `complete-deploy: not running inside a Vercel build, skipping.` when `process.env.VERCEL !== "1"` — Vercel sets that flag during deployments, it's absent locally, so only real deploys can actually mark an announcement live. (3) **Dynamic root OG card at `app/opengraph-image.tsx`.** The previous setup pointed OG + Twitter at the static `/logo-square.png` — a 512×512 square that Discord renders as a small thumbnail beside the link, not a full-width card. Added a per-build 1200×630 `ImageResponse` that layers the square logo (fetched from `/logo-square.png` and embedded as a data URL) next to a "Scrap Mechanic Search Engine" wordmark, a tagline, and a row of kind-tag pills. Twitter card type bumped to `summary_large_image` so Discord, Twitter, and Slack all render the big-image preview. Creation pages keep their own `app/creation/[id]/opengraph-image.tsx`. Dynamic image URLs are content-hashed per build, which cleanly busts Discord's ~24-hour preview cache whenever the artwork changes without needing any manual `?v=N` query-string hack.

- **V9.7 — Live Steam profile refresh + optional kind filter on /admin/triage + worlds thinned alongside tiles + updated logos.** Small bundle of quality-of-life wins shipped together. (1) **Steam profile auto-refresh.** User persona names, avatars, and playtime were only synced to the DB during sign-in, so a user who renamed themselves on Steam kept showing the old name across the site until they signed out and back in. `getCurrentUser` now stamps a new `users.profile_refreshed_at` column (migration 0036) and re-pulls the Steam Web API (`GetPlayerSummaries` + `GetOwnedGames`) when the stamp is null or older than 10 minutes. On refresh it rewrites persona/avatar/profile URL/playtime and stamps the timestamp; on Steam error it silently falls back to the stale snapshot and doesn't stamp so the next request retries. Worst-case one pair of Steam API calls per signed-in user per 10-minute window — well under the 100k/day Steam quota even at scale beyond the current cohort. Everywhere that renders a user's name reads `users.persona_name` at query time, so a single update propagates site-wide. (2) **Optional kind filter on /admin/triage.** New row of pills above the triage stack lets moderators narrow the pending queue to a single creation kind (blueprint / mod / world / tile / …). Each pill shows its total pending count plus a small purple `+N` badge for the community-submitted subset inside that kind, so high-intent rows remain visible even while a filter is active. Community-first ordering is preserved inside the filtered slice — `?kind=mod` still floats community-submitted mods to the top of the stack before auto-ingested ones. Filter state lives in the URL (`?kind=X`) so it survives refresh and can be linked directly; pills are server links rather than client-side transitions so the swipe stack's internal buffer resets cleanly instead of shuffling mid-animation. The "All" pill clears the filter. Per-kind counts come from one extra `GROUP BY kind` query bundled into the existing `Promise.all`; no extra round-trip. (3) **Worlds now thinned to ~5% alongside tiles** on home / `/new` / `/feed.xml`. Worlds have the same large-batch problem as tiles and were crowding out blueprints/mods on mixed listings. `TILE_THIN_CONDITION` renamed to `HIGH_VOLUME_THIN_CONDITION` and extended to `kind IN ('tile', 'world')`. `/tiles`, `/worlds`, `/search`, and any kind-scoped route still return the full catalogue. (4) **Deploy banner upgrades.** Always-on pulse animation (was only under 10s — easy to miss for the first 50). Countdown is now server-synchronized: API returns `serverNow`, client records the offset against its local clock, render math uses the corrected server-perspective time, so every browser on every device sees the same countdown regardless of local clock drift. Offset re-syncs every 8s. (5) **Updated logos.** Creator shipped the long-pending square logo — it's now the site favicon (`app/icon.png`, `app/apple-icon.png`) and the OG/Twitter social-card image (`public/logo-square.png`).

- **V9.5 — Tiles down to 5% + banner stays until new deploy is live + 0033 hotfix.** Three small items shipped together in response to live-site feedback. (1) **Tile thinning tightened from 25% to 5%.** `TILE_THIN_CONDITION` in `lib/db/queries.ts` changed from `abs(hashtext(id)) % 4 = 0` to `% 20 = 0` — still stable per-row and still limited to the home page / `/new` / `/feed.xml`, but only 1 in 20 tiles now shows on those surfaces instead of 1 in 4. Search and `/tiles` still return everything; nothing is deleted from the catalogue. (2) **Deploy banner now sticks until the new bundle is actually live, not on a 30-second timer.** V9.4 auto-hid the banner 30 seconds after the scheduled push moment, which felt abrupt and didn't line up with when users could actually refresh onto new code (Vercel rolling deploys take more than 30 seconds on cold builds). New `deploy_announcements.completed_at` column (migration 0035) is left null by `scripts/deploy.ts` on insert and stamped by a new `scripts/complete-deploy.ts` that runs at the very end of the Vercel build (after `next build` finishes). While completed_at is null the client banner holds "Deploying now — the page will auto-refresh when the new version is ready." indefinitely. Once the row is completed, the client swaps to "✅ New version is live — reloading…" and triggers `window.location.reload()` exactly once per announcement (tracked via `sessionStorage` key `smse_deploy_reloaded_id` so the reloaded page doesn't recursively reload itself for the same announcement). `getActiveDeployAnnouncement()` keeps returning completed rows for a 2-minute tail so slow-polling clients still see the completion and reload. (3) **0033 migration hotfix.** The regenerated V9.3 migration 0033 carried a leftover `DO $$ ... EXCEPTION WHEN duplicate_object ... END $$` block for the `mod_actions` foreign key — a defensive workaround because the 0032 snapshot predated 0031's hand-written additions. The neon-http migrator didn't parse the DO block correctly, the migration failed mid-way in prod, `moderator_since_at` never got added, every `getCurrentUser` call 500'd, and every auth-dependent route (including Steam OpenID return) errored. Rewrote 0033 down to its genuinely-new bits (one `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` plus the audit-log backfill UPDATE). Also added `IF NOT EXISTS` guards on 0034 and 0035 so re-running them on any partially-applied environment is safe.

- **V9.4 — Auto-migrate on build + visitor deploy countdown banner.** Shipped together in response to a production 500 on login: migration `0033` (V9.3's `moderator_since_at` column) hadn't been applied to the prod Neon DB after the V9.3 push, so `getCurrentUser`'s `SELECT *` on `users` tried to read a column that didn't exist and every auth-requiring route 500'd. Two-part fix. (1) **`tsx scripts/migrate.ts` now sits in the build script** (`package.json`) between the existing `fetch-captcha-images` / `fetch-blockdle-data` scripts and `next build`. Every Vercel deploy applies pending migrations before compiling the new code, so a schema change plus its code change can ship together in a single push without the Creator remembering to run `db:migrate` manually. Fail-fast: if a migration breaks, the whole deploy fails and prod stays at the previous working version rather than shipping code that queries a column the DB doesn't have. (2) **New `deploy_announcements` table + `DeployBanner` top bar + `npm run deploy` script** (migration `0034`). `scripts/deploy.ts` inserts a row with `scheduled_at = now() + 60s`, ticks the countdown in the terminal, then runs `git push`. Every visitor on the live site sees a sticky red top-bar countdown during those 60 seconds — `components/DeployBanner.tsx` polls `/api/deploy-announcement` every 8 seconds for the active row and ticks locally at ~30fps so the millisecond display stays smooth without hammering the server. Under 10s it pulses (Tailwind `animate-pulse`). After the scheduled moment it holds "Deploying now — please wait…" for a 30-second grace window then self-hides; the server query drops rows past that tail too, so the banner can't get stuck on an old row. `getActiveDeployAnnouncement()` in `lib/db/queries.ts` is the single read path. Rows are never deleted — the table doubles as a deploy log. Insert runs from local `DATABASE_URL`, same .env loading as `scripts/migrate.ts`, so the script doesn't need a production-only setup.

(4) **Triage bug fixes.** Two issues reported by the site's first moderator. **(a)** The "Steam ↗" link in the `/admin/triage` card header wasn't navigating — `onPointerDown` was calling `setPointerCapture` on the whole card for swipe handling, which hijacked pointer events from interactive children and the anchor never got a proper click. Moved the min-h onto the current card's visible box so the card itself always fills to the viewport floor; parent collapses to current's height and nothing leaks. Next-card sizing via `inset-0` naturally clips to current's bounds.
